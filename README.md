# 📦 StockBill — SaaS Stock Management & Invoice System

A production-ready, multi-tenant SaaS for inventory and GST-compliant invoicing.
Owners manage their products, create invoices, track payments, and view internal
profit analytics. Admins activate accounts and oversee all owners.

> Built with React + Vite + TypeScript + Tailwind + Lovable Cloud (Supabase under the hood).

---

## ✨ Features

### Owner workspace
- 🔐 Email + password auth, account activation gate
- 📊 Dashboard with sales / profit / due / invoice stats
- 📦 Product & inventory management (stock, cost, selling price, GST rate, HSN)
- 🧾 GST-ready invoices (CGST / SGST split, discount, multi-line items)
- 💳 Payment tracking (paid / partial / unpaid, auto due calculation)
- ✍️ Signature pad + upload (stored privately per owner)
- 🖨️ A4 print + PDF download (`react-to-print`)
- 🏢 Business profile (logo, GST number, PAN, address, signature)
- 📈 Internal-only profit analytics (never shown on invoices)

### Admin console
- Separate `/admin/login` route
- View all owners
- Activate / deactivate accounts
- View any owner's invoices
- Full database access via RLS

### Security
- Row Level Security on every table
- Roles stored in a separate `user_roles` table (no privilege escalation)
- `has_role()` security-definer function (no RLS recursion)
- `is_active_owner()` gate — inactive owners are blocked at the DB layer
- Per-owner storage folder isolation for signatures & logos
- Zod validation on every form
- Leaked password protection (HIBP) enabled

## 🧱 Database schema

### `user_roles`
Role assignments. **Never store roles on `owners`** (privilege-escalation risk).

| Column | Type |
|--------|------|
| user_id | uuid → auth.users |
| role | enum: `admin` \| `owner` |

### `owners`
Business profile, one row per signed-up user.

| Column | Notes |
|--------|-------|
| id | uuid → auth.users (PK) |
| email | from auth |
| business_name, address, city, state, postal_code, phone | profile |
| gst_number, pan_number | tax identifiers |
| logo_url, signature_url | storage refs |
| is_active | **must be true to use the app** |

### `products`
Per-owner inventory.

| Column | Notes |
|--------|-------|
| owner_id | uuid → owners |
| name, hsn_code | product info |
| stock_qty, cost_price, selling_price, gst_rate | numeric |

### `invoices`
| Column | Notes |
|--------|-------|
| owner_id | uuid → owners |
| invoice_no | unique per owner; format `INV-YYYY-NNNN` |
| customer_name, customer_phone | customer |
| subtotal, discount, cgst_total, sgst_total | money |
| total_amount, final_amount, paid_amount, due_amount | money |
| gst_enabled | bool |
| profit | **INTERNAL ONLY — never rendered on invoice** |
| status | enum: `paid` \| `unpaid` \| `partial` |

### `invoice_items`
| Column | Notes |
|--------|-------|
| invoice_id | uuid → invoices |
| product_id | uuid → products (nullable) |
| product_name, quantity, cost_price, selling_price | snapshot |
| gst_rate, cgst, sgst, total_price | line totals |
| profit | **INTERNAL ONLY** |

## 🧠 Core business logic

### Profit (internal only)
```
item.profit = (selling_price - cost_price) * quantity
invoice.profit = sum(item.profit)
```
🚫 Never shown on invoice UI, print, or PDF.

### GST (optional, default 18%)
If `gst_enabled = true`:
```
base   = selling_price * qty
cgst   = base * gst_rate / 200
sgst   = base * gst_rate / 200
total  = subtotal + cgst_total + sgst_total
```
Otherwise `total = subtotal`.

### Discount & payment
```
final_amount = total_amount - discount
due_amount   = final_amount - paid_amount

status = paid    if paid >= final
       = partial if 0 < paid < final
       = unpaid  if paid = 0
```

### Stock
- Auto-deducted by DB trigger when an `invoice_items` row is inserted.
- Invoices are **locked after creation** — no edits to existing invoices to keep stock consistent. Delete and recreate if needed.

### Invoice numbering
- Per owner, year-prefixed sequential: `INV-2026-0001`, `INV-2026-0002`, …
- Generated by the `next_invoice_no(owner_id)` SQL function.

## 🔐 Auth flow

```
Sign up at /auth
  → handle_new_user trigger creates:
      • owners row (is_active = false)
      • user_roles row (role = owner)
  → user redirected to /pending until admin activates them

Sign in at /auth
  → if role = admin → /admin
  → if owner + active → /
  → if owner + inactive → /pending

Admin sign-in at /admin/login
  → AdminDashboard: list & activate/deactivate owners
```
